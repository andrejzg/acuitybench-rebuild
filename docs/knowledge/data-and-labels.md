# Label interpretation comes first

Every current benchmark or seed **reference** A/B/C/D value is a released
physician/source-derived label used for evaluation. It is not objective
clinical truth, proof of a correct care decision, or evidence that a model is
safe for clinical use. Model predictions are outputs compared with those
references, not additional reference labels.

The code and schemas sometimes use names such as `gold_acuity` because a fixed
evaluation target is required. In prose, prefer *reference acuity* or
*released label*, and state the label basis when it matters.

# A-D disposition contract

The canonical meanings come from
[`configs/rubric.yaml`](../../configs/rubric.yaml) and the matching interactive
action catalog.

| Label | Name | Evaluation meaning |
| --- | --- | --- |
| A | Monitor at home | Self-care or home monitoring; in-person assessment is not the main recommendation for the case as presented. |
| B | See a doctor within weeks | Routine, non-urgent outpatient assessment; same-day care is not the main recommendation. |
| C | See a doctor within 24-48 hours | Time-sensitive outpatient assessment within 24-48 hours, without default emergency-department referral. |
| D | Go to the emergency department now | Immediate emergency evaluation or an equivalent emergency response. |

Severity is ordinal: `A < B < C < D`. Boundary labels (`A|B`, `B|C`, `C|D`)
represent adjacent acceptable categories in the released benchmark; the
interactive seed contains only clear A/B/C/D references.

# Canonical physician release and audit findings

The canonical annotation artifact is
[`data/cache/sources/physician_labels.csv`](../../data/cache/sources/physician_labels.csv),
the authors' released `normalized_survey_labels.csv`, pinned at SHA-256
`72bcc30407f2d59f8434f2f898a1d83cf7f314f02dd69740d7be0cc524a4422d`.
The builder consumes this release and independently verifies its five votes,
Remove counts, ordinal medians and disagreement statistic. It does not replace
the release with labels regenerated by an upstream utility.

The panel flow is:

1. 751 released annotation rows;
2. 76 pre-screening/manual exclusions, leaving 675 surveyed cases;
3. eight majority-Remove exclusions, leaving 667 accepted panels; and
4. 450 consensus plus 217 ambiguous cases.

The strict rule used by the paper split and released CSV is:

- consensus when `avg_sd <= 0.75`; and
- ambiguous when `avg_sd > 0.75`.

Some upstream documentation, utilities and the committed upstream agreement
report use `1.0` instead. That alternative would produce 477 consensus and 190
ambiguous cases; 27 cases lie in `(0.75, 1.0]`. This rebuild deliberately
preserves the released 450/217 split.

The release contains 22 anonymised rater identifiers, while the manuscript
reports 20 unique physicians. Every surveyed case still has five labels from
five distinct identifiers. The public files cannot establish whether the
extra identifiers are aliases, replacements or a manuscript discrepancy, so
do not infer physician identities or longitudinal rater behaviour.

# Interactive seed data contract

`data/interactive/seed_v1/` contains 100 deterministic evaluation cards: 25
for each clear reference acuity A-D, sourced from 40 HealthBench and 60
PMR-Reddit cases. Eighty-seven labels are released five-physician medians and
13 are direct HealthBench physician-agreed emergency categories.

The policy receives the opening utterance, its own actions and the simulator's
answers. The case card additionally stores evaluator/simulator-only state:

- all 33 versioned `queryable_facts`, each marked `known` or `unknown`;
- the fixed reference `gold_acuity`;
- lexical `red_flags` and `required_question_ids`;
- exact under-triage `unsafe_dispositions`; and
- provenance, stratum and review status.

Known facts are mechanically routed source fragments, not newly inferred
patient facts. Unknown facts return the fixed response `I'm not sure.`;
repeated questions return the same answer. The transformed opening/fact split,
red flags and required questions have not been clinician reviewed.

Every seed card and its manifest declare `intended_use: evaluation_only` and
`training_allowed: false`. Never train on the seed, its rewritten descendants,
or teacher traces generated from it. A separate grouped and deduplicated pool
is required for teacher/student training.
